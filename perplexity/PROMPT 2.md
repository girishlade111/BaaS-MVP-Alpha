You are a senior backend developer. I am building a BaaS platform MVP.

This platform has NO traditional user authentication.
There is no email login, no password, no OAuth, no JWT, no user accounts.

Instead, the platform uses a PROJECT-BASED ACCESS SYSTEM.
Every project has its own unique credentials that act as the identity for that project.

CREDENTIAL STRUCTURE:
- Project Name: a human-readable name the user provides (e.g. "my-app-backend")
- Project ID: an 8-character alphanumeric code auto-generated by the system (e.g. "9F2XK8A3")
  The Project ID acts as the Access Key / Secret Key for that project.

PROJECT CREATION FLOW:
Step 1 — User opens the landing page
Step 2 — User clicks "Get Started"
Step 3 — User is taken to the Project Creation page
Step 4 — User types a Project Name in the input field
Step 5 — User clicks "Create Project"
Step 6 — The system generates a unique 8-character Project ID automatically
Step 7 — A credentials screen appears showing Project Name + Project ID with a warning to save them
Step 8 — User clicks "Go to Dashboard" and enters the project workspace

ACCESSING AN EXISTING PROJECT:
- User clicks "Access Existing Project" on the landing page
- User enters their Project Name and Project ID
- If both match exactly in the system — access is granted
- If either is wrong — access is denied with an error message: "Invalid credentials. Please check your Project Name and Project ID."
- After successful access, credentials are stored in browser sessionStorage

PROJECT ID RULES:
- The Project ID is generated once and can NEVER be changed or regenerated
- The Project ID is shown only once at creation — users are warned clearly to save it
- If a user loses their Project ID, the project becomes permanently inaccessible
- There is no password recovery, no email fallback, no secondary access method
- The Project Name CAN be renamed later from the Project Settings page

SESSION BEHAVIOR:
- Credentials (Project Name + Project ID) are stored in browser sessionStorage after access
- Every API request from the dashboard sends these as HTTP request headers
- If the browser tab is closed, sessionStorage is cleared and the user must re-enter credentials
- There is no "Stay logged in" feature in the MVP
- There are no server-side sessions, cookies, or tokens

SECURITY RULES:
- Every backend API route must validate both Project Name and Project ID before doing anything
- A middleware layer on every dashboard API call checks: does this project_id exist? does the project_name match?
- If validation fails: return 401 Unauthorized
- If validation passes: attach the project context to the request and proceed

EXAMPLE CREDENTIALS:
Project Name: my-app-backend
Project ID (Access Key): 9F2XK8A3

Keep this access system in mind for every backend route, every UI flow, and every dashboard interaction.
